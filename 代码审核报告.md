# 月度绩效考核系统代码审核报告

生成时间：2026-05-11  
审核范围：`assessment-backend`、`assessment-frontend`、当前目录项目文档与构建配置  
审核方式：先分析项目文档与工程结构，再由独立代码审核 agent 审查已完成代码，最后结合本地构建验证与关键代码抽查汇总。

## 一、项目文档与工程概览

当前目录是一个月度绩效考核管理系统，采用前后端分离架构：

- 后端：`assessment-backend`，Spring Boot 3.2.5、Java 17、Spring Security、MyBatis Plus、JWT。
- 前端：`assessment-frontend`，Vue 3、TypeScript、Vite、Pinia、Element Plus。
- 业务文档：根目录存在绩效考核解决方案 PDF/DOCX、抽取文本、原型设计 HTML/PNG，以及 `.qoder/repowiki/zh/content` 下的项目知识库文档。
- `CLAUDE.md` 已描述核心领域模型、角色体系、数据范围、API 响应约定和构建命令。

核心角色包括 `ADMIN`、`FIN_ADMIN`、`DEPT_ADMIN`、`DEPT_LEADER`、`SUPERVISOR`。系统设计上存在数据范围：`ALL`、`UNIT`、`ORG`，并通过前端请求头传递到后端。

## 三、主要问题清单

### P0：前端存在语法错误，当前无法完成生产构建

位置：`assessment-frontend/src/views/dept/IndicatorSet.vue:837`

问题：`editIndicatorRow` 函数结束后出现额外的两个右花括号。

影响：`vue-tsc` 直接失败，前端无法构建和发布。

建议：

- 删除多余的 `}`。
- 修复后重新执行 `npm run build`。
- 将 `npm run build` 纳入提交前或 CI 门禁。

### P1：后端权限边界不足，多个管理接口只要求登录

位置：

- `assessment-backend/src/main/java/com/ccerphr/assessment/config/SecurityConfig.java:28`
- `assessment-backend/src/main/java/com/ccerphr/assessment/controller/SysRoleController.java:60`

问题：`SecurityConfig` 只配置了 `.anyRequest().authenticated()`，控制器层普遍没有 `@RequireRole` 或 `@PreAuthorize`。前端路由 `meta.roles` 只是 UI 限制，不能作为安全边界。

影响：任意已登录用户可以直接调用角色管理、用户管理、权限分配、指标审批、考核组变更等敏感接口。

建议：

- 启用后端方法级授权，例如 `@EnableMethodSecurity` + `@PreAuthorize`。
- 管理接口按角色加权限约束，写接口优先限制到 `ADMIN`、`FIN_ADMIN`。
- 保留前端路由控制作为体验层，但以后端授权为准。

### P1：数据范围完全信任前端 Header，存在越权风险

位置：

- `assessment-frontend/src/api/request.ts:33`
- `assessment-backend/src/main/java/com/ccerphr/assessment/interceptor/DataScopeInterceptor.java:28`
- `assessment-backend/src/main/java/com/ccerphr/assessment/util/DataScopeFilter.java:23`

问题：前端发送 `X-Role-Code`、`X-Data-Scope`、`X-Scope-Id`、`X-User-Id`，后端直接写入 `DataScopeContext` 并用于过滤。

影响：攻击者可篡改 Header 为 `ALL` 或任意 `scopeId`，扩大查询/写入范围，绕过单位和组织隔离。

建议：

- 后端从 JWT 中解析用户 ID，再查询用户有效角色和权限表，服务端计算数据范围。
- 客户端传来的权限 Header 只能用于调试或展示，不能参与授权。
- 对所有使用 `DataScopeFilter` 的查询补充越权测试。

### P1：指标审批角色由请求参数/请求体决定，可伪造审批阶段

位置：

- `assessment-backend/src/main/java/com/ccerphr/assessment/controller/BizIndicatorDefinitionController.java:76`
- `assessment-backend/src/main/java/com/ccerphr/assessment/service/impl/BizIndicatorDefinitionServiceImpl.java:239`

问题：审批列表接收 `roleCode` 查询参数，审批操作从 DTO 读取 `roleCode`，服务端直接用它决定状态流转。

影响：用户可伪造 `roleCode=FIN_ADMIN`，跳过部门负责人或分管领导审批阶段，将指标直接推进到最终审批。

建议：

- 审批角色必须来自服务端认证上下文。
- 审批前校验当前用户是否对指标所属组织、当前状态和审批阶段有权限。
- `nextStatus == null` 不应静默跳过，应返回明确错误。

### P1：自评/他评保存接口信任 DTO 中的组织 ID

位置：

- `assessment-backend/src/main/java/com/ccerphr/assessment/service/impl/BizSelfEvaluationServiceImpl.java:167`
- `assessment-backend/src/main/java/com/ccerphr/assessment/service/impl/BizPeerEvaluationServiceImpl.java:253`

问题：保存自评/他评时直接使用前端传来的 `orgId`、`evaluatorOrgId`、`targetOrgId`、`indicatorId` 查询或创建记录，缺少当前用户和组织、考核组、指标归属关系校验。

影响：部门用户可能构造请求替其他部门保存评分或覆盖草稿。

建议：

- 当前组织和权限范围从服务端认证上下文取得。
- 保存前校验 `examGroupId + orgId + indicatorId` 的合法关系。
- 对评分表建立唯一约束，避免并发重复创建。

### P2：附件上传和删除缺少业务归属校验

位置：

- `assessment-backend/src/main/java/com/ccerphr/assessment/controller/BizAppealController.java:112`
- `assessment-backend/src/main/java/com/ccerphr/assessment/controller/BizAppealController.java:184`

问题：上传只按 `appealId` 插入附件；删除只按 `attachmentId` 删除记录和文件，未校验申诉归属、当前用户权限、申诉状态。

影响：用户可能给他人申诉挂附件，或删除他人附件。

建议：

- 上传/删除前加载申诉与附件，校验当前用户数据范围和申诉状态。
- 文件删除和数据库删除应增加事务或补偿处理。

### P2：评分上限与业务语义不一致

位置：

- `assessment-backend/src/main/java/com/ccerphr/assessment/dto/SelfEvalSaveDTO.java:27`
- `assessment-backend/src/main/java/com/ccerphr/assessment/dto/PeerEvalSaveDTO.java:27`

问题：DTO 允许评分最大 `999.99`，但绩效考核评分通常应限制在 `0-100`，且数据库注释也表达为分数语义。

影响：可提交超过 100 的评分，导致结果计算失真。

建议：

- 将 `@DecimalMax("999.99")` 改为 `@DecimalMax("100")`。
- 服务层也补充业务校验，防止内部调用绕过 Bean Validation。

### P2：登录 Token 角色与前端当前角色可能不一致

位置：

- `assessment-backend/src/main/java/com/ccerphr/assessment/controller/AuthController.java:165`
- `assessment-backend/src/main/java/com/ccerphr/assessment/controller/AuthController.java:193`
- `assessment-backend/src/main/java/com/ccerphr/assessment/security/JwtAuthenticationFilter.java:37`
- `assessment-frontend/src/stores/user.ts:62`

问题：JWT 使用 `user.getRoleCode()` 生成，但返回给前端的 `userInfo.roleCode` 优先取 `availableRoles[0]`。前端切换角色只改 localStorage 和 Header，不重新签发 Token。

影响：如果后端补充角色授权，会出现 JWT 权限、前端显示角色、Header 角色三套来源不一致。

建议：

- 登录时明确选择角色并签入 JWT，或角色切换时重新签发 Token。
- 后端权限判断统一以服务端解析结果为准。

## 四、测试与质量缺口

当前缺少以下自动化测试：

- 后端启动和基础 Controller 测试。
- 权限越权测试：低权限用户访问管理接口。
- 数据范围伪造测试：篡改 `X-Data-Scope`、`X-Scope-Id`。
- 指标审批状态机测试。
- 自评/他评组织归属测试。
- 评分边界测试。
- 前端 `vue-tsc` 和构建测试。

## 五、建议修复优先级

1. 先修复 `IndicatorSet.vue` 的语法错误，确保前端能构建。
2. 统一本机/CI 的 JDK 为 17，避免 Maven 默认走 Java 8。
3. 重构后端鉴权与数据范围计算，不再信任前端 Header。
4. 修复指标审批状态机和审批角色来源。
5. 补齐自评/他评、附件、评分边界等业务校验。
6. 增加最小测试集，把 `mvn clean test` 和 `npm run build` 作为交付门禁。

## 六、总体结论

当前代码已经具备较完整的业务模块雏形，但还不适合直接进入生产或验收环境。最主要风险集中在两个方面：前端构建目前失败，以及后端权限/数据范围边界仍依赖客户端输入。建议先解决构建和鉴权问题，再推进业务细节修复和测试覆盖。
