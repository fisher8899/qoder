## 二、方案设计问题深度剖析

### 🔴 问题一：业务模型设计与代码实现严重脱节

**表现**：
1. 文档定义了清晰的五级角色体系（ADMIN/FIN_ADMIN/DEPT_ADMIN/DEPT_LEADER/SUPERVISOR）
2. 文档定义了明确的数据范围约束（ALL/UNIT/ORG）
3. 代码审核发现 SecurityConfig 只检查登录状态，任意登录用户可访问所有接口

**根本原因**：
- 方案设计时假设了服务端权限检查，但实现时只做了前端路由守卫
- 文档强调"后端必须校验"，但代码中 `@RequireRole` 注解使用不完整
- 安全评审指出的问题在 CHANGELOG 中标记为"已修复"，但代码审核又发现同类问题

**解决方案**：
```java
// SecurityConfig.java 应该这样配置
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**").permitAll()
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/exam-group/*/publish/**").hasAnyRole("ADMIN", "FIN_ADMIN")
    .requestMatchers("/api/exam-group/**").hasAnyRole("ADMIN", "FIN_ADMIN", "DEPT_ADMIN")
    .anyRequest().authenticated()
)
```

**建议**：
1. 立即梳理所有 Controller 接口，按角色矩阵生成权限配置
2. 建立"文档设计 → 代码实现 → 安全测试"的闭环验证
3. 将权限测试纳入 CI 流程

### 🔴 问题二：数据范围设计存在致命漏洞

**文档设计的理想状态**：
```
ADMIN -> dataScope = "ALL" -> 可见全部数据
FIN_ADMIN -> dataScope = "UNIT" -> 可见本单位数据
DEPT_ADMIN -> dataScope = "ORG" -> 可见本组织数据
```

**代码中的实际漏洞**：
```java
// DataScopeContext.java:36-39
public static String getDataScope() {
    DataScopeInfo info = CONTEXT.get();
    return info != null ? info.getDataScope() : "ALL";  // 危险！默认返回全部
}
```

**攻击场景**：
1. 攻击者登录后删除请求头中的 `X-Role-Code`、`X-Data-Scope`、`X-Scope-Id`
2. DataScopeInterceptor 未设置上下文
3. `getDataScope()` 返回默认的 "ALL"
4. 攻击者可见全部数据

**解决方案**：
```java
public static String getDataScope() {
    DataScopeInfo info = CONTEXT.get();
    if (info == null) {
        throw new SecurityException("数据权限上下文未初始化，请检查请求链路");
    }
    return info.getDataScope();
}

// 同时修改 DataScopeInterceptor，在无权限时直接拒绝请求而非返回默认值
```

**建议**：
1. 将"数据范围默认ALL"列为P0级安全事件
2. 在所有使用 DataScopeContext 的地方增加非空检查
3. 添加安全监控，检测异常的数据范围请求

### 🔴 问题三：状态机设计停留在纸面

**文档设计的状态机**：
```
待启动 -> 进行中 -> 指标已发布 -> 考核中 -> 他评中 -> 预发布 -> 已发布
                   ↓             ↓        ↓         ↓
               校验指标审批   校验自评   校验他评   校验复核和申诉
```

**代码中的实际问题**：
```java
// BizExamGroupServiceImpl.java:369-372
public void prePublish(Long id) {
    // 只检查当前状态
    if (!ExamStatus.IN_PROGRESS.getCode().equals(group.getStatus())) {
        throw new BusinessException("...");
    }
    // ❌ 没有校验：指标是否全部审批通过？自评是否完成？他评是否完成？
}
```

**影响**：可以跳过中间步骤直接发布，导致考核结果不可信。

**解决方案**：
```java
@Transactional
public void prePublish(Long id) {
    BizExamGroup group = getById(id);
    if (!ExamStatus.IN_PROGRESS.getCode().equals(group.getStatus())) {
        throw new BusinessException("状态不合法");
    }
    
    // 前置条件校验
    Long examGroupId = id;
    Long unitId = group.getUnitId();
    
    // 1. 校验所有部门指标是否全部审批通过
    long pendingIndicators = indicatorDefinitionMapper.countPendingApproval(examGroupId);
    if (pendingIndicators > 0) {
        throw new BusinessException("存在 " + pendingIndicators + " 个未审批通过的指标，请先完成审批");
    }
    
    // 2. 校验所有部门自评是否提交
    long pendingSelfEval = selfEvalMapper.countPendingSubmit(examGroupId);
    if (pendingSelfEval > 0) {
        throw new BusinessException("存在 " + pendingSelfEval + " 个部门自评未提交");
    }
    
    // 3. 校验所有部门他评是否完成
    long pendingPeerEval = peerEvalMapper.countPendingSubmit(examGroupId);
    if (pendingPeerEval > 0) {
        throw new BusinessException("存在 " + pendingPeerEval + " 个部门他评未完成");
    }
    
    // 4. 校验所有申诉是否已处理
    long pendingAppeals = appealMapper.countPendingHandle(examGroupId);
    if (pendingAppeals > 0) {
        throw new BusinessException("存在 " + pendingAppeals + " 个未处理的申诉");
    }
    
    group.setStatus(ExamStatus.PRE_PUBLISHED.getCode());
    group.setCurrentStep("预发布");
    updateById(group);
}
```

**建议**：
1. 为每个状态转移方法添加前置条件校验
2. 建立状态机配置文件，声明每个状态转移的前置条件
3. 添加状态机可视化工具，便于业务人员理解

## 三、业务流程设计问题

### 🟡 问题六：评分规则存在逻辑漏洞

**文档设计的评分规则**：
```
最终得分 = adminScore（非空）? adminScore : deptScore
```

**代码中的问题**：
```java
// BizReviewScoreServiceImpl.java:429-435
private BigDecimal calcFinalScore(BigDecimal adminScore, BigDecimal deptScore) {
    // 业务规则：优先取 adminScore（管理员打分），如果为空则取 deptScore（他评平均分）
    if (adminScore != null) {
        return adminScore;
    }
    return deptScore;
}
```

**问题分析**：
1. 逻辑正确，但只针对已保存的 `biz_review_score` 记录
2. 如果财务管理员未逐项保存，`submitReview` 时不会为未编辑指标生成最终成绩
3. 结果查询可能缺失部分部门的最终成绩

**解决方案**：
```java
@Transactional
public void submitReview(Long examGroupId, String reviewer) {
    // ... 获取已有记录 ...
    
    // 获取所有已审批指标
    List<BizIndicatorDefinition> approvedIndicators = indicatorDefinitionMapper.selectApprovedIndicators(examGroupId);
    
    // 为每个部门每个指标 upsert 最终成绩
    for (BizExamGroupMember member : members) {
        for (BizIndicatorDefinition indicator : approvedIndicators) {
            // upsert 逻辑：如果记录存在则更新，不存在则创建
            upsertFinalScore(examGroupId, member.getOrgId(), indicator.getId(), reviewer);
        }
    }
    
    generateMonthlyScores(examGroupId);
}
```

### 🟢 问题八：申诉流程与复核流程耦合不足

**文档设计的流程**：
```
申诉提交 -> 等待处理 -> 申诉处理 -> 分数回写 -> 复核重新评估
```

**实际问题**：
1. 申诉处理后只更新了 `biz_appeal` 状态，未触发复核重新评估通知
2. 申诉处理的事务边界不清晰，可能出现"申诉已处理但分数未回写"
3. 缺少申诉处理后的数据一致性校验

**解决方案**：
```java
@Transactional
public void handleAppeal(AppealHandleDTO dto, String handledBy) {
    BizAppeal appeal = getById(dto.getAppealId());
    
    // 1. 更新申诉状态
    appeal.setNewScore(dto.getNewScore());
    appeal.setStatus("HANDLED");
    appeal.setHandledBy(handledBy);
    appeal.setHandledTime(LocalDateTime.now());
    updateById(appeal);
    
    // 2. 回写评分（同一事务）
    updateReviewScore(appeal, dto.getNewScore());
    
    // 3. 发送复核重新评估通知
    sendReevalNotification(appeal);
    
    // 4. 记录审计日志
    auditAppealHandle(appeal, handledBy);
}
```

## 四、技术架构问题

### 🟡 问题九：前后端分离不彻底

**问题表现**：
1. 前端路由守卫依赖角色码，后端 Controller 也依赖角色注解，存在不一致风险
2. `/menu/current` 接口返回的数据与前端路由配置的 roles 不一致
3. 状态管理（Pinia）与后端权限上下文同步机制缺失

**解决方案**：
1. 统一前后端权限模型，由后端提供权威的菜单和权限数据
2. 前端不再硬编码路由 roles，从 `/menu/current` 动态获取
3. 角色切换时清空本地缓存，重新加载菜单

---

### 🟢 问题十：数据库设计存在冗余

**问题表现**：
1. 多个表存储 `org_name`、`unit_name`，但没有外键约束
2. 历史数据修复依赖 `DataFixRunner`，治标不治本
3. 缺少数据完整性校验机制

**解决方案**：
1. 在关键表上添加外键约束
2. 建立数据质量监控，定期检查数据一致性
3. 移除 `DataFixRunner`，从源头保证数据完整性

## 五、项目管理问题

### 🟡 问题十一：文档与代码不同步

**问题表现**：
1. `CHANGELOG.md` 记录"已修复"的 P0 问题，在代码审核中再次发现
2. `DELIVERY_PLAN.md` 定义了退出条件，但未真正执行验证
3. `TEST_STRATEGY.md` 声称有完整测试，实际只有构建验证

**根本原因**：
- 缺乏独立的验证角色
- 文档更新由开发者自己完成，无交叉检查

**解决方案**：
1. 建立"代码完成 → 文档更新 → 独立验证 → 关闭"的闭环
2. 将文档与代码的同步检查纳入代码审核流程
3. 每次代码提交必须更新对应文档，否则拒绝合并

### 🟢 问题十二：风险管控流于形式

**问题表现**：
- `RISK_REGISTER.md` 记录了 8 个风险，但大部分状态仍是"打开"
- 缓解动作描述模糊（如"继续做角色回归"）
- 没有明确的风险关闭条件和验证标准

**解决方案**：
1. 为每个风险定义明确的"关闭条件"
2. 风险责任人必须定期更新状态和验证结果
