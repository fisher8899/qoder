# 项目负责人测试上线冲刺清单

版本日期：2026-05-14  
适用阶段：从“业务功能基本成型”推进到“测试收口、上线交付”

## 一、当前阶段判断

当前项目处于：

`功能开发后期 / 联调收口前 / 尚未进入真正可交付测试阶段`

判断依据：

- 需求文档与原型已定义完整主流程：指标发布、自评、他评、复核、预发布、申诉、正式发布、结果查询与导出。
- 当前代码已具备主链路基础实现，核心业务不是从 0 开始。
- 但服务端权限、数据范围、状态机、评分闭环、申诉回写、测试基线、发布准备尚未完成收口。
- 因此当前状态不是“继续无边界加功能”，而是“尽快收拢范围，转入交付驱动”。

## 二、项目负责人此阶段的核心目标

1. 本轮交付范围明确且冻结，不再失控漂移。
2. 主业务流程可以按角色、按状态、按时序真实跑通。
3. 测试结果可信，能够证明权限、结果、流程没有明显失真。
4. 上线前交付物齐全，发布行为可执行、可回滚、可追责。

## 三、冲刺期间必须盯住的八件事

### 1. 范围冻结

- [ ] 明确本轮上线范围清单
- [ ] 明确不进入本轮的优化项清单
- [ ] 新需求统一进入 `REQUIREMENT_CHANGE_LOG.md`
- [ ] 未评估影响的需求不得直接插入开发

### 2. 以原型时序图为验收基线

- [ ] 按原型时序图拆出完整 UAT 清单
- [ ] 每个角色的可见范围、可执行动作、禁止动作均有验证项
- [ ] 每个流程状态都有进入条件和退出条件
- [ ] 退回、重提、申诉、重评等回路均纳入测试

### 3. 主流程联调切换为业务剧本联调

- [ ] 指标发布到正式发布完整跑通一次
- [ ] 至少执行一轮含申诉回路的完整联调
- [ ] 结果查询、结果明细、导出结果口径一致
- [ ] 联调问题形成清单并按优先级分层

建议剧本：

1. 财务/考核管理员发布月度指标
2. 部门绩效管理员完成自评并提交
3. 考核员/分管领导完成他评并提交
4. 管理员完成复核修正
5. 执行预发布
6. 发起申诉并重新评估
7. 再次复核并正式发布
8. 多角色查询结果并导出

### 4. 阻断测试问题与阻断上线问题分层管理

`测试前必须处理`

- [ ] 主流程关键页面和接口可跑通
- [ ] 状态流转不出现明显乱跳
- [ ] 结果查询和导出具备基础可信度
- [ ] 测试环境可稳定启动

`上线前必须清零`

- [ ] 服务端 RBAC 不完整
- [ ] 数据范围信任前端 Header
- [ ] 审批接口信任前端角色参数
- [ ] 申诉处理不回写评分链路
- [ ] 状态机不完整，可跳步骤推进发布

`可延期但必须记录`

- [ ] 中文乱码治理
- [ ] 局部体验优化
- [ ] 非阻断性结构重构

### 5. 权限与状态机作为红线事项单独盯办

- [ ] 服务端权限来源于后端上下文，不依赖前端传参
- [ ] 数据范围不能伪造
- [ ] 审批、发布、退回、重评等动作均有真实服务端校验
- [ ] 所有状态迁移均校验前置条件和当前状态

### 6. 测试组织提前启动，不等“功能全做完”

- [ ] 已建立主流程冒烟清单
- [ ] 已建立角色权限回归清单
- [ ] 已建立评分/结果口径核对清单
- [ ] 已建立空库初始化或新环境部署验证清单
- [ ] 已建立发布前回归清单

### 7. 提前补齐交付物，不允许最后一周临时拼装

- [ ] 部署说明已整理
- [ ] 环境配置说明已整理
- [ ] 初始化脚本和版本说明已整理
- [ ] 回滚方案已整理
- [ ] 已知问题清单已整理
- [ ] 验收记录模板已准备

### 8. 每日以控制面文档驱动，而不是口头推进

每日必看文档：

1. `PROJECT_PROGRESS.md`
2. `TASK_BOARD.md`
3. `RISK_REGISTER.md`
4. `BUG_FIX_LOG.md`
5. `REQUIREMENT_CHANGE_LOG.md`
6. `RELEASE_CHECKLIST.md`

负责人每日必问：

1. 今天真正完成了什么，不是“做了什么”
2. 当前是否存在阻断测试的问题
3. 当前是否存在阻断上线的问题
4. 下一步是谁在什么时间前完成什么
5. 哪个风险在升级，是否需要决策介入

## 四、项目负责人分阶段执行清单

### A. 未来 3 天内必须完成

- [ ] 冻结本轮上线范围
- [ ] 将原型时序图转成 UAT 场景清单
- [ ] 跑通一轮主流程冒烟
- [ ] 输出阻断测试问题清单
- [ ] 输出阻断上线问题清单
- [ ] 明确每个 P0/P1 的负责人和目标日期

### B. 未来 1 周内必须推进到位

- [ ] 完成一轮完整业务剧本联调
- [ ] 完成一轮权限回归
- [ ] 完成一轮结果口径核对
- [ ] 完成申诉回路验证
- [ ] 完成环境初始化/部署验证
- [ ] 将高风险问题归并到专项整改清单

### C. 测试收口阶段必须满足

- [ ] P0 清零
- [ ] P1 有明确处理结论
- [ ] 主流程回归通过
- [ ] 权限回归通过
- [ ] 结果口径一致
- [ ] 发布检查单通过
- [ ] 风险台账已更新

### D. 上线前最后检查

- [ ] 部署步骤经过实操验证
- [ ] 配置项与环境项已确认
- [ ] 初始化脚本可执行
- [ ] 回滚方案可执行
- [ ] 已知问题已书面确认
- [ ] 验收结论已留档
- [ ] 交付包齐套

## 五、当前最值得亲自盯住的五个问题

1. 本轮需求是否已经冻结，否则测试永远没有边界。
2. 权限和数据范围是否仍由前端“协助决定”，这会直接阻断上线。
3. 状态机和评分链路是否闭环，否则测试结果不可信。
4. 测试是否按业务场景执行，而不是按页面点点点。
5. 发布准备是否提前启动，否则最后阶段必然失控。

## 六、结论

当前项目最需要的不是继续扩写功能，而是把项目从“研发驱动”切到“交付驱动”。

项目负责人此时最有效的管理动作只有三类：

1. 冻结范围
2. 强推场景联调和测试收口
3. 盯死权限、状态机、评分闭环、发布准备这四类红线事项

只要这四类红线没有形成结论，项目就不能被判定为“具备上线交付条件”。
